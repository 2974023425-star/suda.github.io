Spring 核心原理面试笔记（Day2 复盘 · 刘思）

适用：初级 Java 开发面试（目标腾讯）。本笔记汇总 Day2 五道 Spring 核心题的过关口述稿 + 本人踩过的坑，用于每日过一遍防回生。
一句话主线：IOC 容器 → BPP 后置生成 AOP 代理 → 事务=代理前后管事务 → 自动配置按需造 Bean 装容器，五道题串成一条机制链。

Q1 · IOC 与 Bean 生命周期（9 步）

核心要点

  • IOC（控制反转）是思想：对象创建权交给容器；DI（依赖注入）是实现手段。不能说”IOC 分成两部分”。
  • BeanDefinition 是蓝图：启动时注册，不是打在实例上。
  • 生命周期 9 步：注册 BeanDefinition → 实例化（反射 new）→ 属性填充（DI）→ Aware 回调（往已存在对象塞名字，不是造对象）→ BPP 前置 → 初始化（@PostConstruct）→ BPP 后置（AOP 代理在此生成，因为此时 Bean 已是成品）→ 就绪入单例池 → 销毁（@PreDestroy）。

过关口述稿

IOC 把对象创建权交给容器，DI 是注入依赖的手段。容器启动时先把每个 Bean 的定义注册成 BeanDefinition（蓝图），然后实例化（反射 new）→ 属性填充（DI）→ Aware 回调（把名字等塞进已存在的对象）→ BPP 前置 → 初始化（@PostConstruct）→ BPP 后置（在此生成 AOP 代理，因为 Bean 已成成品）→ 入单例池就绪 → 销毁（@PreDestroy）。

本人踩过的坑

  • 把”反射实例化”错安到 Aware 回调那一步。
  • 误以为 IOC = DI 两部分、BeanDefinition 是贴在实例上的注解。

Q2 · 循环依赖与三级缓存

核心要点

  • 三级缓存：singletonObjects（一级·成品池）、earlySingletonObjects（二级·早期引用池）、singletonFactories（三级·工厂池，存 ObjectFactory）。
  • 放工厂 ≠ 调工厂：A 一实例化（半成品）就把 ObjectFactory 放入三级缓存（只是登记）；等 B 查一、二级没有，才去三级缓存调 getObject() 取出 A 的早期引用。
  • 若 A 本身需要 AOP 代理（@Transactional 等），工厂方法当场生成早期代理返回；否则返回原身。目的：保证 B 注入的 A 与最终入一级缓存的成品 A 是同一对象，切面才生效。生成的早期引用/代理还会存二级缓存。
  • 构造器注入的循环依赖无解：构造器实例化必须拿到完整依赖，没有”半成品”阶段，进不了三级缓存，直接抛 BeanCurrentlyInCreationException。

过关口述稿

循环依赖指 A 依赖 B、B 又依赖 A。A 实例化（半成品）后填充 b 触发创建 B，B 实例化后填充 a 时又要 A，而 A 还没初始化完，双方互等就卡住。Spring 用三级缓存解决：A 实例化后就把它的 ObjectFactory 放入三级缓存（此时只是登记）；等 B 查一、二级缓存没有，才去三级缓存调 ObjectFactory.getObject()，当场拿到 A 的早期引用。若 A 本身需要 AOP 代理就生成早期代理返回，否则返回原身——保证 B 注入的 A 与最终入一级缓存的成品 A 是同一对象，切面才生效。生成的早期引用/代理还会存二级缓存。若 A、B 用构造器注入则无解：构造器实例化就必须拿到完整依赖，没有半成品阶段，进不了三级缓存。

本人踩过的坑

  • 用 Maven 类比循环依赖（错：Maven 编译期失败、Spring 运行期可解）。
  • 漏讲三级缓存、漏讲构造器注入无解。
  • 把”放工厂”和”调工厂”混成一步（”生成时调”）。
  • “a 还没初始化”应为”已实例化但未初始化完（半成品）”；”带着代理的原身”应为”生成 A 的早期代理”。

Q3 · AOP 原理（JDK 动态代理 vs CGLIB）

核心要点

  • AOP 本质 = 通过代理在方法调用前后织入增强（Advice：@Before/@After/@Around/@AfterThrowing/@AfterReturning）。代理只给命中切点的 Bean 生成；无切点就是本体。
  • JDK 动态代理：要求目标类实现接口，代理类实现相同接口，经 InvocationHandler 反射调用，只能代理接口里的 public 方法。
  • CGLIB：通过继承目标类生成子类、重写方法拦截（MethodInterceptor），能代理类。
  • 选型：目标有接口用 JDK、无接口用 CGLIB；Spring Boot 2.x 默认强制 CGLIB（proxy-target-class=true）。
  • CGLIB 限制：final 类 / final 方法不能代理（无法继承/重写）、目标需无参构造。

过关口述稿

Spring AOP 底层用代理，两种：JDK 动态代理要求目标类实现接口，代理类实现相同接口、经 InvocationHandler 反射调用，只能代理接口里的 public 方法；CGLIB 通过继承目标类生成子类、重写方法拦截（MethodInterceptor），能代理类。选型：目标有接口用 JDK、无接口用 CGLIB，Spring Boot 2.x 默认强制 CGLIB。CGLIB 限制是 final 类/final 方法不能代理、目标需无参构造；JDK 局限是只能代理接口方法。

本人踩过的坑

  • 选型逻辑臆想成”类本身用 JDK、两个类之间用 CGLIB”（错：真实看目标类是否实现接口）。
  • 误以为 CGLIB 限制是”一个类与多个类做 AOP 不行，因单继承”（错：单继承指代理类已继承目标、不能再继承别的；多切面完全没问题；真正限制是 final 类/final 方法）。
  • “JDK 不能代理 private 因为私有”原因太笼统（准确：JDK 是接口代理，只覆盖接口声明的方法，且代理对象与目标非继承关系，无权访问目标 private）。

Q4 · Spring 事务（传播行为 + 失效场景）

核心要点

  • 事务本质：命中 @Transactional 的 Bean 被换成 AOP 代理（由 BPP 在创建时按需替换、存入容器），外部经代理调方法，代理在前后拿连接（绑 ThreadLocal）→ 提交/回滚，保证 ACID。
  • 为什么用事务：把”扣 A + 加 B”当不可分割单元，要么全成要么全滚，避免数据不一致（转账经典例）。
  • 传播行为（重点记 4 个）：
    • REQUIRED（默认）：有事务加入、没有新建，内外同一事务同生共死。
    • REQUIRES_NEW：挂起当前、新建独立事务，内外互不影响。
    • NESTED：当前事务内开嵌套子事务（savepoint），子可单独回滚、父回滚子也回滚。
    • SUPPORTS：有则加入、无则非事务。
    • （其余：NOT_SUPPORTED 挂起事务非事务跑；NEVER 有事务抛异常；MANDATORY 必须有事务，没有抛异常）
  • 失效场景（高频 6 条）：①同类自调用绕过代理（this.xxx()）；②方法非 public；③异常类型不对（默认只回滚 RuntimeException/Error，checked 不回滚，需 rollbackFor=Exception.class）；④异常被 catch 吞掉；⑤final 方法 CGLIB 无法代理；⑥多线程调用连接不共享（事务绑 ThreadLocal）。
  • 跨类调用（@Autowired B，注入的是 B 的代理）默认 REQUIRED 才”同生共死”；B 用 REQUIRES_NEW 或异常被吞，就不是同时回滚。

过关口述稿

Spring 事务基于 AOP 代理实现 @Transactional。传播行为：REQUIRED 默认，有事务加入、没有新建；REQUIRES_NEW 挂起当前、新建独立事务，内外互不影响；NESTED 在当前事务内开嵌套子事务，子可单独回滚、父回滚子也回滚；SUPPORTS 有则加入、无则非事务。事务失效常见：①同类自调用绕过代理；②方法非 public；③异常类型不对（默认只回滚 RuntimeException，checked 不回滚，需 rollbackFor）；④异常被 catch 吞掉；⑤final 方法 CGLIB 无法代理；⑥多线程调用连接不共享。

本人踩过的坑

  • 起点：事务传播/失效完全不清楚（简历写”熟悉 Spring 事务”，此处是知识缺口，已补齐）。
  • 初期混淆”AOP 只作用于自己类””注入是回调”（错：每个 Bean 各包各的代理；@Autowired 是 DI 属性填充；代理在 Bean 创建时已由 BPP 替换存入容器，注入时直接拿）。
  • 误以为”开启事务就保证两个类同时回滚”（错：仅默认 REQUIRED 同生共死；REQUIRES_NEW 或异常被吞则不是）。

Q5 · Spring Boot 自动配置原理（@Conditional）

核心要点

  • @SpringBootApplication = @Configuration + @ComponentScan + @EnableAutoConfiguration（自动配置开关）。
  • 自动配置类来源文件：META-INF/spring.factories（Spring Boot 2.7 前）/ META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports（2.7+）。里面列了一百多个 XxxAutoConfiguration。
  • 这些类是 @Configuration，内部用 @Bean 定义 RedisTemplate、DataSource 等，靠 @Conditional 按需生效：
    • @ConditionalOnClass：类路径有该类才配（引了 redis 依赖才有 RedisTemplate）。
    • @ConditionalOnMissingBean：容器无用户自定义同类型 Bean 才用默认（约定优于配置）。
    • @ConditionalOnProperty：配了某参数才生效（配了 spring.redis.host 才建连接）。
  • 顺序由 @AutoConfigureBefore / @AutoConfigureAfter 控制。
  • 本质：启动时按清单批量加载自动配置类，用条件注解自动造 Bean 装容器。与 AOP 代理无关（自动配置造的是普通 Bean，无切点即本体）。

过关口述稿

@SpringBootApplication 包含 @EnableAutoConfiguration（自动配置开关）。它借助 AutoConfigurationImportSelector 从 META-INF/spring.factories（2.7 前）/ META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports（2.7+）读取所有自动配置类。这些类是 @Configuration，内部用 @Bean 定义 RedisTemplate、DataSource 等。它们靠 @Conditional 按需生效：@ConditionalOnClass 类路径有该类才配；@ConditionalOnMissingBean 容器无用户自定义 Bean 才用默认；@ConditionalOnProperty 配了某参数才生效。顺序由 @AutoConfigureBefore/After 控制。本质是按依赖和配置自动造 Bean 装容器，与 AOP 代理无关。

本人踩过的坑

  • 把 @SpringBootApplication 当成切点、把自动配置说成”生成代理注入”（错：自动配置是造 Bean + 条件装配，和 AOP 代理完全两码事）。
  • 混淆了”代理”概念的滥用：刚学完 AOP 后见啥都想套代理，需警惕。

机制链一句话总结（面试串讲用）

Spring 启动时用 IOC 容器造 Bean；Bean 初始化后由 BPP 后置给命中切点的 Bean 生成 AOP 代理（JDK/CGLIB）并存入容器；之后谁 @Autowired 谁拿到的是代理——事务 @Transactional 就是靠这个代理在方法前后管连接、保 ACID；而 Spring Boot 自动配置 另走一套 @Conditional 机制，按依赖和配置自动把常用 Bean 造好装进容器，与代理无关。

每日自测清单（合上笔记能讲出来 = 过关）

  • Q1：Bean 生命周期 9 步能脱稿背，且说清 IOC≠DI、BeanDefinition 是蓝图。
  • Q2：三级缓存三层分别是啥、放工厂与调工厂时机、AOP 早期代理生成点、构造器为何无解。
  • Q3：JDK vs CGLIB 区别、选型规则、CGLIB 的 final 限制、JDK 只代理接口方法。
  • Q4：事务本质（代理前后管事务）、REQUIRED/REQUIRES_NEW/NESTED 区别、6 条失效场景。
  • Q5：@EnableAutoConfiguration 是开关、META-INF 清单文件（2.7 前后差异）、@Conditional 三剑客、与 AOP 无关。
