Day 1 学习笔记：HashMap 与集合框架

训练方式：面试官模拟口述 + 逐句挑错，目标是”关上聊天能自己讲顺”
覆盖：Q1 HashMap 底层 / Q2 为什么用 Redis / Q3 ConcurrentHashMap / Q4 ArrayList vs LinkedList + twoSum 算法应用
状态：Q1–Q4 全部过关 ✅，twoSum 理解与代码逻辑全部过关 ✅

〇、前置知识（位运算，卡过关才懂）

概念 记法 说明
异或 ^ 一样得0，不一样得1 用在扰动：h = key.hashCode() ^ (h>>>16)，把高位特征搅进低位少撞桶
与 & 逐位比，同为1才得1 用在寻址：(n-1) & hash，n 是2的幂时等价于 hash % n
无符号右移 >>> 右移N位，左边补0 扰动用 >>> 而非 >>：负数 hashCode 时 >> 会在高位补1污染结果
为什么数组长度必须2的幂 — ① 让 & 等价取余（快）；② 扩容时 e.hash & oldCap 直接拆高低位免重算哈希

自检：42 & 15 = 10（=42%16 ✓）；42 & 9 = 8（≠42%10=2，证明非2的幂不行）。

一、Q1：HashMap 底层原理

过关口述稿

HashMap 底层是 Node 数组 + 链表 + 红黑树。put 时先扰动：h = key.hashCode() ^ (h>>>16)（把高位特征搅进低位，因寻址只认低位，减少冲突）；再用寻址 (n-1) & hash（n 是2的幂，此时 & 等价于取余）定位桶。桶为空直接放；冲突就尾插挂链表，已是红黑树按树规则插。某桶链表 ≥8 且 数组≥64 转红黑树；树节点 ≤6 退化回链表（留7缓冲防7↔8反复横跳抖动）。当 已用桶/容量 > 0.75 时整个数组翻倍并 rehash（扩容时 e.hash & oldCap 免重算）。

易错点（面试必纠）

  • 🔴 不是”链表+红黑树”，最外层是 Node 数组（桶数组）
  • 🔴 扩容和树化是两个独立机制，别焊一起：
    • 扩容触发：已用桶数 / 容量 > 0.75 → 数组×2 + rehash
    • 树化触发：单桶链表≥8 且 数组≥64（否则优先扩容而非树化）
  • 🔴 扰动方向是”高位搅进低位“，不是低位到高位
  • 🔴 扰动用 ^（异或），寻址用 &（与），别混

面试官高频追问

  1. 扩容时元素怎么迁移？答：e.hash & oldCap 拆高低位，原位置 or 原位置+oldCap，免重算
  2. 为什么树化阈值是 8？答：泊松分布，链表到8概率极低（约千万分之六），时空权衡
  3. 为什么负载因子 0.75？答：太接近1冲突剧增，0.75是冲突概率与空间利用率折中

二、Q2：为什么用 Redis 而非本地 Map 当缓存

过关口述稿

本地用 Map 做缓存选 ConcurrentHashMap（线程安全）。普通 HashMap 多线程会出事故：JDK 1.7 头插法并发扩容链表成环 → get 死循环 CPU 100%；JDK 1.8 尾插解决成环，但并发 put 仍数据覆盖/节点丢失。ConcurrentHashMap 1.8 用 CAS + synchronized 锁单桶保证原子。
但即便用 ConcurrentHashMap，本地缓存也不是最优——单 JVM 一份、多实例不共享会缓存不一致；JVM 重启就丢不能持久化；要自己写 TTL 淘汰。所以项目用 Redis：所有实例共享、独立进程可持久化、自带过期+LRU、不占 JVM 堆。

易错点

  • 🔴 “扩容/树化影响性能”不是选 Redis 的真正理由（Redis 自身也是哈希表也要 rehash）
  • 🔴 最大理由 = 多实例共享 + 持久化 + 淘汰策略 + 不占堆，微服务多实例下本地 Map 缓存不一致是死穴
  • 🟡 “Redis 保证线程安全”不严谨 → 应说本地改用 ConcurrentHashMap 即可，真正选 Redis 是上面那几条
  • 🟡 “Redis 在 JVM 里面”口误 → Redis 是独立服务，所有实例连同一个

三、Q3：ConcurrentHashMap 1.7 vs 1.8

过关口述稿

CHM 1.8 线程安全：读无锁；写时 CAS 优先（桶空 CAS 直接插无锁），有竞争才 synchronized 锁住该桶头节点。相比 1.7：1.7 用 Segment 数组，每段一把 ReentrantLock，默认 16 段=16把锁，并发上限16、热点段串行；1.8 把锁粒度降到单个桶，并发度≈桶数量、不同桶互不互斥，冲突更少并发更高（synchronized 经 JVM 锁升级优化后性能足够）。普通 HashMap 并发不能用，正是因为它无同步——1.7 成环死循环、1.8 数据覆盖丢失，CHM 用 CAS/锁变原子才安全。

易错点

  • 🔴 “改进在哪”= 锁粒度从段降到桶（锁整间房 → 锁一个柜子），并发度 16 → ≈桶数
  • 🔴 段数 16 是 concurrencyLevel 默认、取2的幂便于寻址、构造时定死不能扩（1.7硬伤）；1.8 桶可翻倍增长故并发能放大
  • 🟡 不是”一律 synchronized”：桶空时 CAS 无锁，有竞争才锁

四、Q4：ArrayList vs LinkedList

过关口述稿

ArrayList 底层数组，有容量，满了扩容1.5倍并搬移；支持随机访问 O(1) 所以查快，但中间插删要搬移元素慢。LinkedList 底层双向链表，无需扩容（加元素就新建节点），增删只需断链 O(1) 所以增删快，但按索引访问要从头遍历查慢。项目里列表操作用 ArrayList 为主（悦电设备列表、SudaFlow 商品列表都是遍历+随机取，ArrayList 最合适）；LinkedList 只在频繁头尾增删才考虑，实际业务少。

易错点

  • 🔴 题目问”各自怎么扩容”——LinkedList 不需要扩容（无定容，加元素 new 节点）
  • 🔴 漏答”项目用过哪个、为什么”——诚实答 ArrayList 为主，因遍历+随机访问 O(1)
  • 🟡 LinkedList 按索引删 = 先遍历找到（慢）再断链 O(1)，慢在”找”不在”删”

五、twoSum 算法（HashMap 应用）

四个理解问题（全对）

  1. 为什么想到哈希表：暴力法对每个数都从头扫补数 O(n)，哈希表把查找降 O(1)，整体 O(n²)→O(n)
  2. 为什么用 map：数组只能存值，map 能绑”值→下标”两个信息
  3. map 存什么：遍历过的元素值 + 它的下标
  4. key/value：key=元素值，value=该元素下标

代码逻辑过关稿

循环里先查后放：每轮先算 temp = target - nums[i]，containsKey(temp) 查 map；查到说明之前遍历过的元素是补数，返回 [map.get(temp), i]；查不到再把 nums[i] 和下标 i 放进 map。必须先查后放，否则当前元素被自己当补数、同元素用两次。边遍历边放（非先建完再查）好处：① 天然防同元素复用 ② 命中即 return 可提前结束。

易错点

  • 🔴 顺序是先查后放，不是先放（先放后查会同一元素用两次，如 nums=[4,2,1] target=8 误返 [0,0]）
  • 🔴 “先建完再查 vs 边遍历边放”区别不是时间复杂度（都是 O(n)），是真·防复用 + 提前返回
  • 🟡 例子走查 nums=[2,7,11,15] target=9：i=0 查7无→放{2:0}；i=1 查2命中→return [0,1]，7 不会被放进 map

  • 扩容（0.75、翻倍、rehash、e.hash&oldCap 免重算）
  • 树化（≥8 且 ≥64）/ 退化（≤6，留7缓冲）
  • HashMap 1.7 头插成环死循环 / 1.8 数据覆盖
  • 为什么本地缓存选 ConcurrentHashMap、为什么还用 Redis
